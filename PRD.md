# PRD — Персональный digital-garden про vibe-coding и agentic engineering

> Статус: draft (готов к обсуждению / переносу в GitHub Issue)
> Язык контента сайта: русский
> Тип проекта: личный knowledge-base + публичный блог

---

## Problem Statement

Я хочу систематически каталогизировать и развивать собственные знания по теме vibe-coding и agentic engineering (статьи, концепции, Cursor-skills, MCP-серверы, актуальные тренды), и одновременно использовать сам процесс ведения этого ресурса как практическую площадку для AI-driven разработки.

Существующие решения не подходят:
- **Notion / Obsidian** — закрытые, не публичные, не показывают git-historу как часть бренда, не интегрированы с GitHub Actions / Cursor agents.
- **Готовые блог-платформы (Medium, Habr, Substack)** — нельзя применить кастомный дизайн уровня `DESIGN.md`, нет встроенного механизма курируемых GitHub-трендов, нельзя писать через Cursor + git.
- **Шаблонные SSG-блоги** — обычно либо слишком «продуктовые», либо не имеют встроенной системы для трендов / каталогов skills и MCP, либо требуют CMS/auth/БД.

В итоге мне нужен сайт, который:
1. Является **читаемым публичным артефактом** (не приватная заметка в Obsidian).
2. Имеет **строгий брутализм-дизайн** в духе xAI (`DESIGN.md`).
3. Полностью **редактируется через git+MDX** — то есть поддаётся Cursor-агентам как первый класс.
4. Делает **GitHub-тренды в нашей нише первоклассным контентом**, а не виджетом-сбоку.
5. Не несёт инфраструктуру, которая мне не нужна (БД, auth, CMS, runtime AI).

## Solution

Next.js-сайт с дизайн-системой по `DESIGN.md`, контентом в виде MDX-файлов в репозитории, и пятью контент-коллекциями (articles, concepts, skills, MCP, trends). Тренды собираются ежедневным GitHub Action и коммитятся в репо как JSON-снимки — что превращает историю трендов в самостоятельный временной артефакт.

Никакого runtime-AI на сайте: AI работает только в авторском workflow (Cursor + git-агенты пишут MDX через PR). Никакой БД: всё либо в MDX, либо в JSON в репо. Никаких комментов, аккаунтов, email. Архитектура портабельна по хостингу — собирается в standalone Node-процесс, запускается в любом Docker.

В MVP открыты UI для articles, concepts и trends. Skills и MCP уже определены в схеме контент-слоя, но их UI поднимается во второй фазе. Это позволяет начать наполнять данные без блока на доделку всех страниц.

## Implementation Decisions

### Стек

- **Фреймворк**: Next.js 15 (App Router), TypeScript strict (`noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`).
- **Стили**: Tailwind v4 с CSS-токенами через `@theme`, реализующими палитру и шкалы из `DESIGN.md`.
- **Контент-слой**: Velite + Zod-схемы для пяти коллекций. На билде Velite валидирует frontmatter, типизирует индексы, сериализует MDX в RSC-формат.
- **Подсветка кода**: Shiki через `rehype-pretty-code`, кастом-тема, переопределяющая токены до почти-монохромного варианта (белый с разной opacity для keywords/strings/comments) — чтобы не нарушать брутализм.
- **Шрифты**: Geist Sans + GeistMono из npm `geist` через `next/font/local`. (Заменяет `universalSans` из `DESIGN.md` — он проприетарный для xAI и не лицензирован.)
- **Linter/Formatter**: Biome (заменяет ESLint+Prettier одной утилитой).
- **Pre-commit**: Husky + lint-staged. Применяется существующий скилл `setup-pre-commit` из `.agents/skills/`.
- **Тесты**: Vitest для модульных тестов; E2E не входят в MVP.
- **CI/CD**: GitHub Actions — workflow для PR-проверок и второй workflow для ежедневного фетчинга трендов.
- **Хостинг**: отложен до фазы 2 (разработка локально). Архитектура хост-агностична через `output: 'standalone'`.
- **БД**: отсутствует. Контент в MDX, тренды в JSON.
- **Лицензии**: MIT (код), CC BY 4.0 (контент).

### Контент-коллекции (Velite)

Пять коллекций определены в `velite.config.ts` с Zod-схемами. У каждой свой набор обязательных и опциональных frontmatter-полей; общие поля: `title`, `description`, `slug`, `tags[]`, `publishedAt`, `updatedAt?`, `draft: boolean`.

- **articles** — длинные материалы. Дополнительно: `cover?`, `series?`, `readingTime` (вычисляемое).
- **concepts** — короткие энциклопедические записи. Дополнительно: `aliases[]?`, `seeAlso[]?`.
- **skills** — описания Cursor-skills (название, триггер, инструкции). Дополнительно: `trigger`, `inputs?`. UI откладывается на фазу 2, схема активна с MVP.
- **mcp** — описания MCP-серверов (название, инструменты, ссылки). Дополнительно: `tools[]`, `repo`, `language`. UI откладывается на фазу 2.
- **trends-meta** — лёгкая модель для отображения JSON-снимков, не MDX-коллекция; читается напрямую из `content/trends/*.json`.

### Роутинг

- `/` — лента всех опубликованных материалов (articles + concepts), последние трендовые репо в виджете.
- `/articles`, `/articles/[slug]`
- `/concepts`, `/concepts/[slug]`
- `/skills`, `/skills/[slug]` — фаза 2.
- `/mcp`, `/mcp/[slug]` — фаза 2.
- `/trends` — текущий снимок (`latest.json`).
- `/trends/[date]` — исторический снимок (`YYYY-MM-DD.json`).
- `/tags/[tag]` — кросс-коллекционный листинг.
- `/about` — автор.
- `/colophon` — стек, принципы, история проекта (мета-страница).
- `/feed.xml`, `/sitemap.xml`, `/robots.txt` — служебные.

### Trends pipeline

GitHub Actions cron запускается раз в сутки. Скрипт-фетчер (модуль **Trends Fetcher**) принимает:
1. `seed.yml` — список тематических `topic:`-фильтров и whitelist-репозиториев («watch_repos»).
2. GitHub-клиент (REST + Search API) — внешняя зависимость, инжектируется для тестируемости.
3. Текущее время — для фильтра «pushed в последние N дней».

Фетчер:
- Запрашивает GitHub Search для каждого topic с фильтрами: `pushed:>now-7d`, `stars:>min_stars`, исключая archived и forks.
- Для каждого `watch_repo` делает прямой запрос (даже если он не попадает под topics).
- Дедуплицирует по `owner/repo`.
- Нормализует поля: имя, owner, описание, язык, звёзды (всего, дельта за неделю), topics, last_pushed.
- Ранжирует по конфигурируемой формуле (по умолчанию — дельта звёзд за неделю с лёгкой доплатой за recency).
- Записывает `content/trends/YYYY-MM-DD.json` и `content/trends/latest.json`.
- Делает git commit + push, что триггерит автодеплой (когда хостинг будет настроен).

При сбое (например, rate-limit) — Action падает с понятным сообщением; следующий запуск восстанавливает.

### Стартовый seed для трендов

Закоммичен как `scripts/trends-seed.yml`. Содержит начальный набор topics (agents, ai-agents, mcp, llm, coding-agent, prompt-engineering, langchain, и т.д.) и watchlist (anthropics/claude-code, modelcontextprotocol/*, getcursor/cursor, Aider-AI/aider, cline/cline, block/goose, continuedev/continue, langchain-ai/*, vercel/ai, microsoft/autogen и др.). Расширяется PR-ами по мере находок.

### Дизайн-система

Реализуется как набор примитивов в `src/components/`, отражающих токены `DESIGN.md`:

- **Button** (Primary, Ghost, Text-link)
- **Tag / MonoTag**
- **Card / BorderedCard**
- **Heading** (Display, Section, Body)
- **Container**
- **Nav / Header / Footer**
- **CodeBlock** (обёртка над Shiki-выходом для одинаковых отступов и опции copy)

Все примитивы — без shadow, с 0px-радиусом по умолчанию, цветовая палитра — только белый/тёмный с opacity-вариациями.

**Правка `DESIGN.md`**: упоминания `universalSans` заменяются на `Geist Sans`. Метрики (размеры, веса, line-heights) остаются.

### MDX-компоненты

Кастомный mapping для базовых тегов: `h1..h6` (с anchor-ссылками), `a` (внешние ссылки → новый таб + индикатор), `code/pre` (через Shiki), `blockquote`, `ul/ol`, `img` (next/image с lazy-loading). Доступен как minimal MDXComponents в провайдере.

### Engagement / SEO

- RSS/Atom — генерируется на билде из `articles` + `concepts` (только опубликованные, не drafts).
- Sitemap — генерируется из всех публичных страниц + всех документов всех коллекций.
- robots.txt — allow по умолчанию.
- OG-картинки — через `@vercel/og` (используется как библиотека, не как Vercel-фича). Шаблон рисует заголовок в GeistMono на тёмном фоне + плашка коллекции в нижнем углу. Кэшируется в `public/og/` или генерируется в едж-ранрайме на любом хосте.

### Не реализуется в MVP, но архитектурно поддержано

- Pagefind-поиск (post-build шаг)
- Wiki-links + backlinks (remark-плагин + Velite-резолвер)
- Email-подписка (Server Action + outbound API)
- Аналитика (script-tag слот в layout)

## Testing Decisions

### Что считается хорошим тестом

Тест проверяет **внешне наблюдаемое поведение** модуля, а не его внутреннюю реализацию. Хороший тест:

- Описывает контракт модуля в терминах входа/выхода или эффектов, а не внутренних шагов.
- Не ломается при рефакторинге, если контракт сохранён.
- Не использует фейковые сетевые / файловые операции, если можно инжектировать зависимость.
- Имеет имя, читаемое как утверждение о поведении.

Тестам подлежат **deep modules** с pure-функциональными интерфейсами (`(input) → output`). UI-компоненты и Next-страницы тестам не подлежат: они тонкие и быстро меняются.

### Модули, покрываемые тестами в MVP

1. **Trends Fetcher** — главный объект тестирования.
   - GitHub-клиент инжектируется как зависимость и мокается.
   - Тестируется: парсинг seed.yml, формирование запросов, нормализация ответов, дедуп между topics-результатами и watchlist'ом, ранжирование, запись JSON.
   - Кейсы: пустой результат, частичный сбой одного запроса, дубликаты репо в разных топиках, watchlist-репо без topic-совпадения, archived/fork исключаются.

2. **Tag Index Builder** — pure-функция от массива документов.
   - Тестируется: корректное группирование, сортировка по количеству, нормализация регистра, обработка коллекционных пересечений.

3. **Feed Generators** (RSS, Atom, sitemap, robots).
   - Snapshot-тесты на стабильность XML-выхода + проверка на парсимость через `fast-xml-parser`.
   - Кейсы: drafts исключаются, updatedAt используется когда задан, корректный pubDate.

4. **Velite Zod-схемы**.
   - Кейсы: валидный frontmatter проходит, невалидный (отсутствующий title, неверный date, неизвестный enum) даёт читаемую ошибку.
   - По одному «valid» и нескольким «invalid» кейсам на каждую коллекцию.

5. **MDX Renderer** (Shiki + плагины + components map).
   - Snapshot-тесты на 4–5 типичных конструкций: заголовок с anchor, code-block с языком, ссылка внешняя/внутренняя, callout, изображение.
   - Не тестируются классы Tailwind напрямую — только структура DOM и наличие ключевых атрибутов.

### Прецеденты

В репозитории пока нет тестов. Эти модульные тесты на Vitest задают паттерн, к которому будем подтягивать будущие модули. Существует скилл `tdd` в `.agents/skills/` — применяется при добавлении новых deep-модулей в фазе 2 (например, Pagefind index builder, Backlinks resolver).

## Out of Scope

Эти пункты осознанно **не входят в MVP** и помечены как таковые, чтобы избежать scope creep:

- **Хостинг и деплой**. Решение отложено до завершения MVP. Разработка ведётся локально. Архитектура портабельна.
- **Runtime AI на сайте** — никаких чатов, RAG, эмбеддингов, AI-сводок. AI присутствует только как часть авторского workflow (Cursor + git-агенты).
- **Комментарии** в любой форме (Giscus, Disqus, кастомные).
- **Аккаунты и авторизация**. Нет user-таблицы, нет login.
- **CMS / admin UI** (Sanity, Notion, Keystatic). Источник истины — git.
- **Email-подписка** в MVP. (Архитектурно подготовлена, но не реализуется.)
- **i18n / EN-перевод**. Сайт RU-only.
- **Pagefind / любой полнотекстовый поиск** — фаза 2.
- **UI для коллекций skills и mcp** — фаза 2. Схема активна с MVP, чтобы данные уже можно было копить.
- **Wiki-links и build-time backlinks** — отложено, может вернуться.
- **E2E / визуальные регрессионные тесты** (Playwright) — оверкилл для статического блога с одним автором.
- **Аналитика** (Plausible / Yandex.Metrica / Umami) — решение откладывается.
- **Социальный автопостинг** (Telegram, X, LinkedIn) — нет цели роста аудитории.
- **БД** любого вида (Postgres, SQLite, Redis, KV) — не нужна.
- **Vercel-specific фичи** (`unstable_after`, Vercel KV/Blob) — намеренно избегаются ради портабельности.
