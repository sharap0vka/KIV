# Коммит, пуш и закрытие issue

Если пользователь просит: "сделай коммит, запушь с upstream, прочитай текущий issue и закрой его по критериям", используй этот процесс.

## Правила безопасности

- Не коммить секреты (`.env`, ключи, credentials) без явного подтверждения пользователя.
- Не использовать destructive git-команды (`reset --hard`, force push), если пользователь явно не попросил.
- Не закрывать issue, если acceptance criteria не подтверждены.

## Процесс

### 1) Проверить статус и контекст

Выполни:

```bash
git status
git diff
git log --oneline -n 10
```

Если есть неоднозначность, уточни у пользователя scope коммита.

### 2) Сделать commit

1. Добавь релевантные файлы в stage.
2. Составь короткий message про "зачем".
3. Выполни commit.
4. Проверь, что рабочее дерево чистое или остались только нерелевантные изменения.

### 3) Сделать push с upstream

Определи, есть ли tracking branch:

```bash
git rev-parse --abbrev-ref --symbolic-full-name @{u}
```

- Если upstream уже есть: `git push`
- Если upstream не настроен: `git push -u origin HEAD`

### 4) Прочитать текущий issue

Определи issue одним из способов:

1. Если пользователь дал номер/URL — используй его.
2. Иначе найди issue в названии ветки/последнем commit (например, `#123`).
3. Если issue не найден — спроси номер у пользователя и остановись.

После определения issue:

```bash
gh issue view <number>
gh issue view <number> --comments
```

### 5) Проверить критерии перед закрытием

Собери checklist:

- [ ] Линт/тайпчек/тесты для затронутого кода проходят (если применимо в проекте)
- [ ] Все acceptance criteria выполнены
- [ ] Нет открытых блокеров в комментариях
- [ ] Изменения запушены в remote
- [ ] Пользователь не просил оставить issue открытым

Если хотя бы один пункт не подтвержден — не закрывай issue и явно напиши, что не выполнено.

### 6) Закрыть issue

Если checklist полностью выполнен:

```bash
gh issue close <number> --comment "Готово: все критерии приёмки выполнены, изменения запушены."
```

## Формат ответа пользователю

Отвечай коротким отчетом:

1. Commit SHA и сообщение
2. Push результат (и был ли настроен upstream)
3. Issue номер и краткий статус критериев
4. Закрыт issue или нет (и почему)
