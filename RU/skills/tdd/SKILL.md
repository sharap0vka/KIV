# Разработка через тесты (TDD)

## Философия

**Ключевой принцип**: тесты должны проверять поведение через публичные интерфейсы, а не детали реализации. Код может полностью поменяться, тесты - нет.

**Хорошие тесты** близки к интеграционным: они проходят реальный путь кода через публичные API. Они описывают _что_ делает система, а не _как_ она это делает. Хороший тест читается как спецификация - "пользователь может оформить заказ с валидной корзиной" явно отражает функциональность. Такие тесты переживают рефакторинги, потому что не завязаны на внутреннюю структуру.

**Плохие тесты** жестко связаны с реализацией. Они мокают внутренние зависимости, тестируют приватные методы или проверяют обходным путем (например, запросом к БД напрямую вместо использования интерфейса). Сигнал проблемы: тесты падают после рефакторинга, хотя поведение не изменилось. Если переименование внутренней функции ломает тест, значит тест проверял реализацию, а не поведение.

См. [tests.md](tests.md) для примеров и [mocking.md](mocking.md) для правил мокирования.

## Антипаттерн: горизонтальные срезы

**НЕ пиши сначала все тесты, а потом всю реализацию.** Это "горизонтальный срез" - когда RED трактуется как "написать все тесты", а GREEN как "написать весь код".

Это приводит к **плохим тестам**:

- Массово написанные тесты проверяют _воображаемое_ поведение, а не _реальное_
- Начинаешь тестировать _форму_ (структуры данных, сигнатуры функций), а не пользовательское поведение
- Тесты теряют чувствительность к реальным изменениям: проходят при сломанном поведении и падают при рабочем
- Ты слишком рано фиксируешь структуру тестов, еще не понимая реализацию

**Правильный подход**: вертикальные срезы через tracer bullets. Один тест -> одна реализация -> повтор. Каждый следующий тест учитывает, что стало понятно в предыдущем цикле. Поскольку код только что написан, ты точно понимаешь, какое поведение важно и как его проверить.

```
НЕПРАВИЛЬНО (горизонтально):
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

ПРАВИЛЬНО (вертикально):
  RED->GREEN: test1->impl1
  RED->GREEN: test2->impl2
  RED->GREEN: test3->impl3
  ...
```

## Процесс

### 1. Планирование

Перед написанием кода:

- [ ] Согласуй с пользователем, какие изменения интерфейса нужны
- [ ] Согласуй, какие поведения тестировать в приоритете
- [ ] Найди возможности для [deep modules](deep-modules.md) (узкий интерфейс, глубокая реализация)
- [ ] Спроектируй интерфейсы с учетом [testability](interface-design.md)
- [ ] Составь список проверяемых поведений (не шагов реализации)
- [ ] Получи подтверждение плана от пользователя

Спроси: "Как должен выглядеть публичный интерфейс? Какие поведения наиболее важно протестировать?"

**Нельзя протестировать все.** Уточни с пользователем, какие поведения критичнее всего. Сфокусируйся на критических путях и сложной логике, а не на каждом возможном edge case.

### 2. Tracer bullet

Напиши ОДИН тест, который подтверждает ОДНУ вещь о системе:

```
RED:   Пишем тест для первого поведения -> тест падает
GREEN: Пишем минимальный код -> тест проходит
```

Это твой tracer bullet - доказательство, что путь работает end-to-end.

### 3. Инкрементальный цикл

Для каждого следующего поведения:

```
RED:   Пишем следующий тест -> падает
GREEN: Минимальный код -> проходит
```

Правила:

- Один тест за раз
- Только столько кода, сколько нужно для текущего теста
- Не закладывайся заранее под будущие тесты
- Держи фокус тестов на наблюдаемом поведении

### 4. Рефакторинг

После прохождения всех тестов ищи [кандидатов на рефакторинг](refactoring.md):

- [ ] Устрани дублирование
- [ ] Углубляй модули (прячь сложность за простыми интерфейсами)
- [ ] Применяй принципы SOLID там, где это естественно
- [ ] Анализируй, что новый код показывает о существующем коде
- [ ] Запускай тесты после каждого шага рефакторинга

**Никогда не рефактори в RED.** Сначала выйди в GREEN.

## Чеклист цикла

```
[ ] Тест описывает поведение, а не реализацию
[ ] Тест использует только публичный интерфейс
[ ] Тест переживет внутренний рефакторинг
[ ] Код минимален для текущего теста
[ ] Не добавлены спекулятивные фичи
```
