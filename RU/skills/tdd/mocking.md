# Когда использовать моки

Мокай только на **границах системы**:

- Внешние API (платежи, email и т.д.)
- Базы данных (иногда - предпочтительнее тестовая БД)
- Время и случайность
- Файловая система (иногда)

Не мокай:

- Свои собственные классы/модули
- Внутренние зависимости между компонентами
- То, что ты полностью контролируешь

## Проектирование под мокирование

На границах системы проектируй интерфейсы так, чтобы их было легко мокать:

**1. Используй внедрение зависимостей**

Передавай внешние зависимости извне, а не создавай их внутри:

```typescript
// Легко замокать
function processPayment(order, paymentClient) {
  return paymentClient.charge(order.total);
}

// Трудно замокать
function processPayment(order) {
  const client = new StripeClient(process.env.STRIPE_KEY);
  return client.charge(order.total);
}
```

**2. Предпочитай SDK-подобные интерфейсы вместо универсального fetch**

Создавай отдельные функции под каждую внешнюю операцию вместо одной общей функции с условной логикой:

```typescript
// ХОРОШО: каждую функцию можно мокать независимо
const api = {
  getUser: (id) => fetch(`/users/${id}`),
  getOrders: (userId) => fetch(`/users/${userId}/orders`),
  createOrder: (data) => fetch('/orders', { method: 'POST', body: data }),
};

// ПЛОХО: для мока нужна условная логика внутри мока
const api = {
  fetch: (endpoint, options) => fetch(endpoint, options),
};
```

Плюсы SDK-подхода:
- Каждый мок возвращает одну конкретную форму данных
- Нет условной логики в тестовом setup
- Проще видеть, какие endpoint'ы затрагивает тест
- Типобезопасность отдельно для каждого endpoint
