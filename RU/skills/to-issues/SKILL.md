# В задачи (Issues)

Разбей план на независимые GitHub issues с помощью вертикальных срезов (tracer bullets).

## Процесс

### 1. Собери контекст

Используй уже доступный контекст из диалога. Если пользователь передал номер или URL GitHub issue, получи его через `gh issue view <number>` (включая комментарии).

### 2. Изучи кодовую базу (опционально)

Если кодовая база еще не изучена, сделай это, чтобы понять текущее состояние проекта.

### 3. Сформируй вертикальные срезы

Разбей план на **tracer bullet** issues. Каждый issue - это тонкий вертикальный срез, проходящий через ВСЕ интеграционные слои end-to-end, а НЕ горизонтальный срез одного слоя.

Срезы могут быть типа `HITL` или `AFK`. `HITL` требует участия человека (например, архитектурного решения или дизайн-ревью). `AFK` можно реализовать и смержить без участия человека. По возможности предпочитай `AFK`.

<vertical-slice-rules>
- Каждый срез дает узкий, но ПОЛНЫЙ путь через все слои (schema, API, UI, tests)
- Завершенный срез можно показать или проверить самостоятельно
- Предпочитай много тонких срезов вместо нескольких толстых
</vertical-slice-rules>

### 4. Проведи сверку с пользователем

Покажи предлагаемую декомпозицию нумерованным списком. Для каждого среза укажи:

- **Title**: короткое понятное название
- **Type**: HITL / AFK
- **Blocked by**: какие срезы (если есть) должны быть завершены раньше
- **User stories covered**: какие пользовательские истории покрываются (если они есть в исходных материалах)

Спроси пользователя:

- Подходит ли уровень детализации? (слишком крупно / слишком мелко)
- Верно ли определены зависимости?
- Нужно ли какие-то срезы объединить или дополнительно разделить?
- Верно ли проставлены типы HITL и AFK?

Итерируйся, пока пользователь не подтвердит структуру.

### 5. Создай GitHub issues

Для каждого утвержденного среза создай GitHub issue через `gh issue create`. Используй шаблон ниже.

Создавай issues в порядке зависимостей (сначала блокеры), чтобы в поле "Blocked by" ссылаться на реальные номера задач.

<issue-template>
## Parent

#<parent-issue-number> (если источник - GitHub issue, иначе секцию опусти)

## What to build

Краткое описание вертикального среза. Описывай end-to-end поведение, а не послойную реализацию.

## Acceptance criteria

- [ ] Criterion 1
- [ ] Criterion 2
- [ ] Criterion 3

## Blocked by

- Blocked by #<issue-number> (если есть)

Или "None - can start immediately", если блокеров нет.

</issue-template>

Не закрывай и не изменяй родительский issue.
